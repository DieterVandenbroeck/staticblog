Telenet announces more usability, the internet screams at the lack of security

Dieter Vandenbroeck | 12 May 2015
"

In March, Telenet (a Belgian telecom operator) proudly announced that its customers would only require a single username-password combination to gain access to 6 key services. Probably expecting their customers to praise this initiative, the internet started to question this broadened attack surface and, more importantly, the (crappy) password policy Telenet enforces.

Designing an authentication mechanism based on a username-password combination that is easy to use on a wide variety of systems and that offers an appropriate level of security is hard. However, there are some basic rules that should be followed nevertheless, including complexity requirements (in terms of length and characters used), limited reuse of passwords and limiting the possibilities of brute-force attacks. Using Telenet’s #telenettipt used to present the ‘1 password for 6 applications’ idea, I want to point out their 6 fails with 1 password:

  • The minimum length for the password is 6 characters. Although best practice dictates a password of at least 8 characters (Owasp), I presume the additional 2 characters that need to be provided on a tablet (probably only once with a ‘remember me’ functionality, until a password change) really are too big a trouble.
  • The maximum length for the password is 8 characters. Although there is a reason to limit the length (from a certain point on, additional length is only taking up server resources while waiting for the remainder), a maximum length equal to the best practice minimum length gives people the shivers.
  • There is only one limit on reusing my password: although I can reuse whatever password I have ever used (that meets the requirements), I cannot change my password to my current password.
  • Limit the authentication scope / entry points: when multiple entry points (e.g. applications) are permitted, it’s hard to synchronize the requirements for the password. The password requirements enforced by the application installed on a tablet should be equal to the requirements on a website. If the requirements are set in a fat client (e.g. tablet application), an update of the requirements without every user updating all applications will bring many problems to the end-users.
  • Although I did not test if there are checks present to counter brute-force attacks, at first sight this kind of controls seems to be missing. With an expanded scope (refer to #4), it is of uttermost importance that all authentication attempts are handled by the same back-end (instance), so that brute-force attacks can be detected as soon as possible (regardless of the entry point).
  • Access to one application gives access to all other applications: because only a single password is used, and application-specific passwords are not, it’s necessary to update your password on every device (which in my case is already 4) or application (even more) after a change. Because doing this is really annoying and cumbersome, it almost makes me choose a password that is ridiculously easy to remember / type in.
