When security through obscurity is a good idea

Dieter Vandenbroeck | 18 Aug 2013
"

Usually, the principle of security through obscurity is just utterly wrong. The idea behind this principle is to have obscurity in your solution as the principal means of security. Of course, this is not a valid substitute for real security: once your obscurity schema is broken, your security would be broken. It might form a defense against automated scripts, but an attacker with sufficient determination will always be able to fight through this defense.

But as with many things, there is something good to say about this as well. Let's take WordPress as an example. In 2012, 60 million websites were powered by WordPress according to Forbes; 37.73% of websites using a CMS are powered by WordPress according to BuiltWith. This is a huge amount of websites that share (partially) the same code running everything. If there is a vulnerability in this code, there is an enormous amount of websites at risk...

By default, WordPress installations will use the 'wp_' prefix for all database tables. Put this and information a vulnerability in the WordPress core together, and you've got a decent part of the internet (which part can be checked easily) vulnerable to something that can be (easily) exploited by an automated script. What is the easy solution for an automated attack? Right, obscurity! Changing all the table names away from the default (e.g. remove or edit the prefix) will already help you on this one.

What about file / directory names? If you put your administration panel in the subdirectory 'admin', every automated tool and every manual guess will disclose the location of your administrative core. The same goes for moving it an obscure directory that will not be found automatically, but including it in the 'robots.txt' file. While you really need to have strong security on that core of your website, making it harder to find lead unskilled / undetermined eyes to another target.
